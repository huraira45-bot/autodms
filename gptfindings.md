# GPT Findings

Date: 2026-06-12

## Scope

This file summarizes the codebase read-through findings for the dealership DMS project in this repository. No source files were changed during the read-through.

Generated/build/dependency artifacts were not manually reviewed line by line, including `node_modules`, `dist`, `.xlsx`, `.bak`, and generated DBML/table dump files. Source, config, docs, routes, controllers, services, utilities, tests, migrations, and setup scripts were reviewed at the code level.

## High-Level Architecture

The project is a dealership management system built on top of a large legacy SQL Server database.

- Backend: Node.js/Express in `Software/`.
- Frontend: Vite/React in `Software/frontend/`.
- Database: legacy schema dump at root plus additive DMS migrations in `Software/migrations/`.
- Main backend entry point: `Software/server.js`.
- Main database connection: `Software/config/db.js`.
- Main RBAC/module registry: `Software/config/modules.js`.

The backend is the primary business-logic layer. The frontend is mostly operational screens that call backend APIs directly through Axios.

## Backend Findings

The backend covers these major domains:

- Authentication and module-based permissions.
- Workshop job cards, labour, parts issue, sublet work, accessories, care-off discounts, and damage marks.
- Procurement and inventory: GRN, GRTN, store sales, and store sale returns.
- Finance: chart of accounts, vouchers, payments, POS settlement, reports, party ledger, system accounts, and tax rates.
- CRO/CRD: complaints, follow-ups, surveys, reminders, campaigns, KYC flags, inquiries, escalation, WhatsApp/Twilio integration, and reports.
- Vehicle sales: catalog, inquiries, bookings, documents, payments, allocation, master invoice, delivery/gate pass, cancellation, incentives, and GL posting.

Important backend files and patterns:

- `Software/server.js` wires public routes, auth middleware, protected API routes, static frontend serving, and cron jobs.
- `Software/middleware/auth.js` validates Bearer JWT tokens and attaches `req.user`.
- `Software/controllers/finalizeController.js` centralizes finalization and unfinalization workflows.
- Accounting journal creation is mostly isolated in pure builder services under `Software/services/`.
- Operational posting services insert finance vouchers and party-ledger entries inside caller-owned SQL transactions.

## Auth And Permissions

- Login uses bcrypt-only password verification.
- Older documentation mentions legacy password comparison, but the current code rejects non-bcrypt stored hashes.
- JWTs include user, group, employee, and module information.
- Backend route/module checks are the real security boundary.
- Some frontend routes only hide or show pages client-side, so backend gates matter most.

## Finalization And Accounting

Finalization is a key system boundary.

Supported finalization domains include:

- `JOBCARD`
- `GRN`
- `GRTN`
- `STORE_SALE`
- `SSR`
- `VOUCHER`

Operational records are finalized by setting `IsFinalized = 1` and posting accounting vouchers. Manual draft vouchers are finalized by changing voucher status from Draft to Posted.

Unfinalization uses an approval workflow. For posted vouchers, unfinalization creates reversal vouchers rather than simply reopening the original voucher.

Downstream-reference checks prevent unsafe reopening when related payments, returns, allocations, or dependent records already exist.

## Inventory And Workshop

Workshop job cards are one of the central business flows.

- RO numbers are generated by job type.
- Care-off discount caps are enforced.
- Labour, parts, sublet, taxes, accessories, and damage marks are stored with job-card context.
- Parts issue snapshots GST and landed cost.
- Finalized job cards are mutation-blocked.
- Job-card finalization can trigger CRD follow-up, survey, and reminder flows.

Inventory posting uses landed-cost and tax snapshots for GRN/GRTN/store-sale/SSR flows. Opening-stock history has correction scripts, so inventory movement assumptions should be handled carefully.

## CRO And CRD

The CRO/CRD module is broad and mature compared with a simple ticketing layer.

It includes:

- Complaint lifecycle and escalation.
- CRD follow-up creation after finalized job cards.
- Public survey tokens and survey capture.
- Service reminders and reminder cron.
- Campaign segmentation/execution.
- WhatsApp/Twilio inbound and outbound handling.
- KYC flags by chassis/customer.
- Reports for SLA, survey scores, reminders, campaigns, touchpoints, escalation heatmaps, repeats, and verdict tracking.

Twilio runs in stub/safe mode unless Twilio environment variables are configured.

## Sales Module

The sales module handles vehicle lifecycle and customer booking flows.

Major areas:

- Vehicle models, variants, and vehicle stock.
- Product-info inquiries from CRO into sales.
- Booking creation and state transitions.
- Required documents and payment proofs.
- Allocation and open-allocation memo support.
- Master invoice and delivery/gate-pass flow.
- Partial-delivery approval.
- Cancellation approval chain.
- Incentive policies, accruals, and disbursement.
- GL posting for booking payment, master invoice, delivery, and incentive transactions.

Some GL posting paths intentionally skip posting if required system-account mappings are missing. Migration `033` appears to provide optional default mappings.

## Frontend Findings

The frontend is a Vite/React single-page app.

Important files:

- `Software/frontend/src/main.jsx`
- `Software/frontend/src/App.jsx`
- `Software/frontend/src/context/AuthContext.jsx`
- `Software/frontend/src/demoMode.js`
- `Software/frontend/src/index.css`

Patterns:

- Axios is used directly from pages/components.
- Auth token is stored in localStorage under `dms_token`.
- The production API base is relative unless `VITE_API_URL` is set.
- Dev default API is `http://localhost:5000`.
- Demo mode can fake login and return empty/success responses.
- Pages are operational screens rather than a marketing app.

Large or central pages include:

- Job card form.
- Booking detail.
- CRO reports.
- CRD follow-ups.
- Complaint detail.
- Receive/make payment.
- POS settlement.

## Database And Migrations

The root database dump describes a much larger legacy ERP-style database with hundreds of tables. The active DMS app uses a focused dealership subset plus newer `dms_*` tables.

Migration themes:

- Accounting foundation and system accounts.
- Tax rates and tax snapshots.
- Party ledger and voucher status/reversal fields.
- Finalization flags for operational documents.
- CRO/CRD tables and follow-up/complaint bridge.
- Sales catalog, bookings, payments, documents, negotiation, incentives, cancellation, and sales GL.

The migration folder is more reliable than older prose documentation for newer DMS behavior.

## Tests

The test suite mainly covers pure business/accounting logic.

Covered areas include:

- Care-off discount helpers.
- Escalation timing.
- Reminder service logic.
- Journal builders for job card, GRN, GRTN, store sale, SSR, and payments.

There is limited or no automated coverage for full database-integrated controller/posting flows.

## Notable Risks Or Gaps

- Some docs are stale compared with the current code.
- Auth is bcrypt-only despite older notes suggesting legacy comparison.
- Database-integrated workflows have less automated test coverage than journal builders.
- Sales GL posting can be skipped when system accounts are unmapped.
- Gross-margin reporting may have a bug: discount appears defined but not accumulated.
- Sales incentive reversal appears to mark accruals reversed without posting a financial reversal voucher.
- Inventory opening-stock history has correction scripts, so inventory valuation logic should be treated carefully.
- Frontend module visibility is useful UX, but backend permission checks are the real enforcement layer.

## Practical Starting Points

For backend work, start from the route, then controller, then service or journal builder:

- Routes: `Software/routes/`
- Controllers: `Software/controllers/`
- Services: `Software/services/`
- Migrations: `Software/migrations/`

For frontend work, start from:

- App routing: `Software/frontend/src/App.jsx`
- Auth: `Software/frontend/src/context/AuthContext.jsx`
- API base/demo behavior: `Software/frontend/src/main.jsx` and `Software/frontend/src/demoMode.js`
- Page components: `Software/frontend/src/pages/`

For accounting changes, inspect the relevant journal builder first, then the posting service, then the finalization controller.
